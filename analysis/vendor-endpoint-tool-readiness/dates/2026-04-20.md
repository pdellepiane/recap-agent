## Observations

- The swagger file defines two vendor operations only:
  - `GET /api-web/vendor/filtered/full` (`getFilteredVendorsFull`)
  - `POST /api-web/vendor/quote` (`createVendorQuote`)
- The search endpoint has enforceable query constraints (enums and regex patterns) that map well to strict tool schemas.
- The quote endpoint requires `eventDate` and documents `today_or_future` behavior.
- Live validation confirms the date rule:
  - yesterday returns an `eventDate` validation error;
  - today removes that date error while leaving unrelated `benefitId` validation error.
- Paolo's note aligns with observed behavior: hardcoding date to today's value in tool layer avoids asking users for extra data and prevents deterministic validation failures from past dates.
- Coverage check confirms `filtered/full` adds richer descriptive and promo payloads, but not better structured location detail in live data:
  - `info.description`: 100% in `filtered/full` vs 0% in legacy.
  - `promos` presence: 60.2% in `filtered/full` vs 0% in legacy.
  - `location.city/state/country`: 0% in `filtered/full` snapshot, while legacy endpoint had any location signal in 61.3% of records.
