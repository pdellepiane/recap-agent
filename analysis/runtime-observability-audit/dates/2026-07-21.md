CloudWatch Logs Insights queries aggregated:

- completion events by `message.outcome` and `message.status_code`;
- successful requests by channel, node, and delivery action;
- repeated successful `message.message_id_hash` values;
- application `ERROR` events.

The DynamoDB scan projected timestamps, channels, timing phases, token totals,
cache rate, classifier fields, quality flags, invariants, node, tool names,
provider-result counts, persistence status, and identifiers used only for local
deduplication. Raw user and assistant text was not exported.

A second WhatsApp-only pass computed SHA-256 of each DynamoDB `message_id`
in-memory, removed the raw identifier, and joined it to CloudWatch's
`message.message_id_hash`. The join compared counts, timestamps, user hashes,
nodes, delivery actions, durations, trace IDs, plan/conversation identity, and
assistant-message hashes. Only 12-character hash prefixes were retained in the
supporting artifact for duplicate/error groups.

## Observations

- The current runtime code hash matched the local artifact, but the current
  runtime build had no post-deployment request.
- Structured request logs revealed two application 500s that did not appear in
  the native Lambda `Errors` metric.
- The successful-turn store contained high-quality typed evidence for 139 recent
  turns, including timing, tokens, classifier decisions, invariants, tools, and
  automated quality flags.
- Three duplicate message identifiers produced two successful turns each.
- Provider sync completed its latest weekly refresh; knowledge sync was disabled
  and stale relative to its checked-in template.
- There were no application alarms, active X-Ray traces, or consistent log
  retention across the three functions.
- The perf schema stores raw user previews and raw channel message identifiers.
- All 133 structured WhatsApp successes had matching DynamoDB records and agreed
  on hashed user and final node. The single DynamoDB-only success preceded the
  deployment of structured completion logging.
- Both application 500s recovered through same-message-ID retries.
- Two duplicated IDs returned two distinct assistant replies each; one of those
  races created separate plans and conversations from a single inbound message.

## Findings Promoted

- Deployment alignment, request health, successful-turn telemetry, observability
