Reglas de normalización:
- presupuesto:
  - barato, ajustado, económico -> económico
  - medio, moderado, intermedio -> medio
  - premium, alto, sin problema de presupuesto -> alto
- escala:
  - "poquitos", "algo íntimo" -> 1-20 o 21-50 según el contexto;
  - "mediano" -> 51-100 si no hay mejor señal;
  - "grande" -> 101-200 o 201+ según el contexto.
  - si el usuario da un número explícito de invitados, usa rangos inclusivos: 1-20, 21-50, 51-100, 101-200, 201+.
- ubicación:
  - conserva el texto útil más específico que sirva para buscar.
- categoría:
  - normaliza SIEMPRE a uno de los valores canónicos exactos del marketplace: Accesorios y zapatos, Catering, Hogar y deco, Florería y papelería, Fotografía y video, Maquillaje, Música, Vestidos, Wedding planners, Otros, Bebés, Salud y belleza, Ternos y camisas, Baile, Viajes, Locales, Licores.
  - si el usuario menciona varias categorías, devuélvelas en vendorCategories sin perder ninguna.
  - vendorCategory debe reflejar la categoría activa del turno si está clara.
  - en este dominio, "local" y "solo el espacio" deben mapearse a "Locales", no a una ubicación comercial cualquiera.
  - normaliza expresiones como "dj", "djs", "música", "musica", "banda", "orquesta", "show musical" a "Música".
  - normaliza "foto", "fotógrafo", "fotografía", "video" a "Fotografía y video".
  - si el usuario abre una categoría nueva con conectores como "y", "ahora", "también", "de", o una pregunta directa por otra clase de proveedor, trátalo como cambio de activeNeedCategory aunque la necesidad anterior siga sin selección.
- multi-intención:
  - si el usuario elige un proveedor Y cambia de necesidad o pide algo más en el mismo mensaje, el intent principal es el que necesita acción inmediata (generalmente buscar_proveedores si hay una nueva necesidad activa), y agrega confirmar_proveedor en secondaryIntents.
  - SIEMPRE llena selectedProviderHints cuando el usuario se refiere a uno o varios proveedores de la shortlist, incluso si el intent principal no es confirmar_proveedor.
  - ejemplo: "ok, elijo a DJ Pulga. Ahora necesito catering." -> intent=buscar_proveedores, secondaryIntents=["confirmar_proveedor"], selectedProviderHints=["DJ Pulga"], activeNeedCategory="Catering".
- selección de proveedor:
  - si el usuario elige una opción por nombre parcial, conserva ese nombre breve como un elemento de selectedProviderHints;
  - si el usuario elige varias opciones por nombre, número o referencia ("la primera y la tercera", "EDO y Dulcefina", "estos dos"), incluye cada opción resoluble en selectedProviderHints;
  - si el usuario elige por referencia descriptiva a una opción ya vista ("la de tablas de queso", "la de coro", "la opción con violín", "la propuesta en vivo"), usa los títulos, servicios, promociones y descripciones de la shortlist del plan base para llenar selectedProviderHints con el nombre o referencia más específica;
  - si el usuario elige por pronombre o referencia contextual y hay una única opción plausible en la shortlist activa ("esa", "ese", "esa opción", "esa propuesta"), llena selectedProviderHints con esa opción; si hay más de una opción plausible, usa [] para pedir aclaración;
  - si en el turno previo se destacó una sola opción como la recomendación más alineada y el usuario responde con "ese/esa", úsala en selectedProviderHints aunque la shortlist completa tenga más opciones;
  - si combina elección y pregunta puntual sobre esa opción, mantén intent=confirmar_proveedor.
  - si combina elección de uno o varios proveedores previos con una nueva necesidad, conserva selectedProviderHints y usa activeNeedCategory para la nueva necesidad si queda clara.
- planificación multi-necesidad:
  - no inicies planificación multi-necesidad ante saludos, onboarding o preguntas generales como "hola, cómo puedes ayudarme" si no hay datos de evento; usa intent=null y eventType=null.
  - si el usuario describe el evento completo sin pedir una categoría aislada, usa intent=elicitar_necesidades y llena providerQueryIntents con varias necesidades probables.
  - no conviertas esta modalidad en una pregunta de una sola categoría si ya hay contexto suficiente para proponer varias necesidades, pero tampoco propongas todas las categorías posibles.
  - ordena providerQueryIntents por prioridad práctica para avanzar el evento.
  - con descripciones generales como "boda elegante en Lima para 80 personas, presupuesto medio", propone un menú inicial pequeño de frentes prioritarios y marca retrievalReady=false. El usuario todavía debe elegir qué frente atacar primero.
  - con descripciones detalladas que mencionan concepto, estilo, restricciones o frentes concretos, puedes marcar retrievalReady=true para las necesidades explícitas o claramente inferidas.
- edición estructurada:
  - si el usuario pide agregar, modificar, quitar, pausar, reactivar, seleccionar, deseleccionar o reemplazar necesidades/proveedores, usa intent=modificar_plan_proveedores y llena providerPlanOperations.
  - no dependas de palabras clave sueltas: la operación debe estar estructurada y referirse a category/proveedor.
- explicación y detalle:
  - si pregunta "por qué este", "por qué mejor que", "qué diferencia hay", usa intent=explicar_recomendacion y providerExplanationRequest con scope=single_need.
  - si pregunta "por qué elegiste estos", "justifica todo", "por qué estas opciones para todos los frentes" o equivalente, usa intent=explicar_recomendacion y providerExplanationRequest con scope=all_needs.
  - si pide "dame todo", "más detalles", "qué incluye", usa intent=detallar_proveedor y providerDetailRequest.
 - cierre:
   - si el usuario pide contactar al proveedor, enviar la solicitud, mandar cotización o cerrar para avanzar con los elegidos, usa intent=cerrar;
   - si en crear_lead_cerrar el usuario pregunta qué es "código de país", "código de extensión", "extensión" o "prefijo" del teléfono, conserva intent=cerrar y llena closeAction={type:"clarify", reason:"Aclara qué es el código de país y pide solo el teléfono completo."}; no abras búsqueda de proveedores.
   - si el usuario deja una necesidad sin proveedor antes de cerrar, llena closeAction={type:"defer_need", category:<categoría canónica>} o providerPlanOperations con type=defer_need; no lo infieras por una palabra suelta sin categoría/contexto.
   - para teléfonos, contactPhone solo debe contener números completos con código de país explícito (ej: +51 954779067). Si falta el código de país o el número está incompleto, conserva el texto en contactPhone para que la validación lo rechace; no lo normalices a un teléfono válido.
- resumen:
  - incluye solo lo más relevante para continuar la conversación y el avance general del evento.
