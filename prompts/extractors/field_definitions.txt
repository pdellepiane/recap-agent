Definiciones de campos:

- intent:
  - elicitar_necesidades: el usuario describe un evento completo o pide ayuda para armar el plan; debes convertirlo en varias necesidades de proveedores con providerQueryIntents.
  - No uses elicitar_necesidades para saludos, preguntas de onboarding o "cómo puedes ayudarme" sin datos de evento. En esos casos deja intent=null o usa el intent específico si hay una pregunta real.
  - buscar_proveedores: quiere iniciar o continuar una búsqueda con intención de ver opciones.
  - refinar_busqueda: quiere ajustar criterios, ampliar o acotar resultados, incluyendo pedir más opciones dentro de la misma necesidad.
  - ver_opciones: quiere revisar alternativas sin expresar refinamiento claro.
  - confirmar_proveedor: quiere elegir un proveedor o pedir avanzar con una opción específica.
  - modificar_plan_proveedores: quiere agregar, cambiar, eliminar, pausar, reactivar, seleccionar, deseleccionar o reemplazar necesidades/proveedores del plan. Usa providerPlanOperations.
  - explicar_recomendacion: quiere entender por qué un proveedor conviene más que otro. Usa providerExplanationRequest.
  - detallar_proveedor: quiere toda la información disponible de un proveedor. Usa providerDetailRequest.
  - retomar_plan: quiere seguir desde un plan ya guardado.
  - cerrar: quiere cerrar el plan o avanzar a contactar/cotizar con proveedores ya elegidos.
  - pausar: quiere dejarlo para después o retomarlo luego.
  - consultar_faq: quiere hacer una pregunta general sobre Sin Envolturas, precios, comisiones, funcionalidades, pagos, listas de regalo, soporte, capacidades de la plataforma, reclamos de productos, compras del Shop, o cualquier tema de la base de conocimiento que no requiera identificar un evento real del usuario. También úsalo cuando el usuario pida algo fuera de alcance pero relacionado con Sin Envolturas, por ejemplo diseñar una web externa o pedir contacto humano por un error.
  - consultar_evento_invitado: quiere consultar información de un evento real de Sin Envolturas asociado al usuario que pregunta, ya sea como asistente, anfitrión, celebrado, dueño del evento o comprador. Úsalo para preguntas sobre fecha, hora, nombre del evento, RSVP, confirmados, invitados, acompañante, anfitrión, rol del usuario, evento propio, evento donde es host/celebrado, regalos u órdenes de ese evento. Si dice que tiene un problema con "mi evento", que no puede compartir su evento, o pregunta cómo ver o revisar datos de "mi evento", "mis invitados", "los confirmados" o "todo en un solo lugar", es consultar_evento_invitado, no un saludo ni onboarding genérico. Ejemplos: "tengo un problema con mi evento", "no puedo compartir mi evento", "¿a qué hora es mi evento?", "¿confirmé asistencia?", "¿dónde veo los confirmados?", "¿puedo llevar acompañante?", "¿cuál es la boda de Ana?", "¿qué evento tengo en Sin Envolturas?".

- secondaryIntents:
  - lista opcional de ints adicionales presentes en el mensaje, además del intent principal.
  - Úsalo cuando el mensaje combina intenciones claras. Ejemplo: "ok, elijo a DJ Pulga. Ahora necesito catering para tortas" tiene intent=buscar_proveedores y secondaryIntents=["confirmar_proveedor"] porque el usuario está cambiando de necesidad pero también seleccionó un proveedor de la shortlist.
  - Si el usuario elige uno o varios proveedores Y pide algo más (otra categoría, otra búsqueda, cerrar), incluye confirmar_proveedor en secondaryIntents y llena selectedProviderHints con cada referencia elegida.
  - Puede estar vacío si el mensaje tiene una sola intención clara.

- intentConfidence:
  - alto si el mensaje es directo;
  - medio si hay inferencia razonable;
  - bajo si hay ambigüedad.

- eventType:
  - tipo de evento canónico o null.
  - DEBE ser null si el usuario no describe ningún evento o solo saluda/pregunta cómo puedes ayudar.
  - DEBE ser exactamente uno de estos valores cuando sí hay evento: boda, cumpleanos, corporativo, baby_shower, graduacion, bautizo, aniversario, quinceanos, otro.
  - Usa "otro" solo cuando hay un evento real descrito pero no encaja en los tipos conocidos. No uses "otro" como fallback para ausencia de evento.
  - Usa razonamiento semántico, no solo palabras exactas: "matrimonio" y "wedding" -> boda; "cumple" -> cumpleanos; "15 años" -> quinceanos.

- vendorCategory:
  - necesidad activa o principal del turno actual.
  - DEBE ser exactamente uno de estos valores canónicos: Accesorios y zapatos, Bebés, Catering, Florería y papelería, Fotografía y video, Hogar y deco, Locales, Licores, Maquillaje, Música, Baile, Salud y belleza, Ternos y camisas, Vestidos, Viajes, Wedding planners, Otros. Si el usuario dice "decoración", usa "Hogar y deco". Si dice "flores", usa "Florería y papelería". Si dice "dj" o "música", usa "Música". Si pide "audiovisuales" o "producción audiovisual", usa "Fotografía y video". Nunca inventar valores fuera de esta lista.
  - si el usuario nombra una nueva categoría mientras había otra necesidad activa en el plan, úsala como vendorCategory del turno actual.

- vendorCategories:
  - lista de necesidades de proveedores mencionadas en este turno o claramente vigentes.
  - cada elemento DEBE ser uno de los valores canónicos listados en vendorCategory.
  - incluye categorías futuras o paralelas si el usuario las nombra como parte del plan.

- activeNeedCategory:
  - categoría con la que el usuario quiere avanzar ahora, usando solo los valores canónicos de vendorCategory.
  - si el usuario cambia de tema dentro del mismo plan hacia otro proveedor (ejemplo: "y qué djs tienes"), actualízala a esa nueva categoría aunque no haya cerrado la necesidad anterior.
  - si el usuario menciona varias categorías pero no prioriza ninguna, déjalo en null.

- location:
  - ciudad, distrito, zona o país útil para la búsqueda.

- budgetSignal:
  - señal textual o semántica de presupuesto, incluso si no hay monto exacto.

- guestRange:
  - normaliza a uno de estos valores: 1-20, 21-50, 51-100, 101-200, 201+, unknown.
  - si hay un número explícito de invitados, usa límites inclusivos al asignar el rango.

- preferences:
  - gustos o atributos deseables, no obligatorios.

- hardConstraints:
  - restricciones obligatorias que sí cambian la selección.

- assumptions:
  - inferencias razonables que conviene transparentar después.

- conversationSummary:
  - resumen corto, acumulativo y actualizado del plan del evento y del frente activo.

- selectedProviderHints:
  - lista de índices, nombres parciales o referencias claras a proveedores ya recomendados.
  - si el usuario elige una o varias opciones desde una shortlist visible y la referencia es resoluble con el plan base, incluye cada referencia como un elemento.
  - captura selecciones plurales como "la primera y la tercera", "EDO y Dulcefina" o "estos dos".
  - no reemplaces nombres de proveedores desconocidos por nombres parecidos del plan. Si el usuario escribe un nombre que no coincide con seguridad, conserva el texto literal como hint o deja la referencia sin resolver para pedir aclaración.
  - usa [] solo si la referencia sigue siendo genuinamente ambigua incluso con la shortlist del plan base.

- pauseRequested:
  - true si el usuario quiere pausar o retomar más tarde.
  - false si quiere cerrar definitivamente (esos casos van en intent: cerrar).

- contactName:
  - nombre completo del usuario cuando lo proporciona explícitamente para el cierre del plan.
  - null si no se menciona en este turno.

- contactEmail:
  - dirección de correo electrónico del usuario cuando la proporciona para el cierre.
  - null si no se menciona en este turno.

- contactPhone:
  - número de teléfono del usuario cuando lo proporciona para el cierre.
  - si el usuario envía solo un número de teléfono mientras estamos pidiendo datos de contacto, extráelo como contactPhone aunque no haya otra intención explícita.
  - el número debe tener al menos 6 dígitos; si es más corto, no lo extraigas.
  - null si no se menciona en este turno.

- providerFitCriteria:
  - objeto obligatorio con los criterios estructurados que se usarán para reordenar proveedores antes de responder.
  - eventType: el mismo valor canónico de eventType; null si no hay señal.
  - needCategory: necesidad activa normalizada en lenguaje natural breve, por ejemplo "catering", "fotografía", "música" o "local"; null si no hay señal.
  - location: ubicación útil para la búsqueda; null si no hay señal.
  - budgetAmount: monto numérico si el usuario lo dijo; null si no hay monto.
  - budgetCurrency: PEN o USD si se puede inferir; null si no hay moneda.
  - mustHave: atributos deseables o necesarios que deben favorecer a un proveedor.
  - shouldAvoid: señales que deben penalizar a un proveedor.
  - rankingNotes: una frase corta en español que explique qué debe privilegiar el reranking.

- providerQueryIntents:
  - lista estructurada de necesidades de proveedores que deben consultarse en paralelo cuando el usuario describe un evento completo.
  - cada elemento debe incluir category canónica, label, priority, queries, preferences, hardConstraints, missingFields, retrievalReady y fitCriteria.
  - queries es una lista plana de consultas buscables para esa necesidad de proveedor. Cada query debe incluir id, label, category, queryStrings, mustHave, shouldAvoid, maxSelections y allowCrossCategory.
  - Consolida primero: si varias frases apuntan al mismo tipo de proveedor y el mismo entregable, usa una sola query con varias queryStrings o mustHave.
  - Separa queries solo cuando los componentes dentro de la misma necesidad realmente podrían requerir proveedores distintos o evidencia claramente distinta. Ejemplo: Catering con sushi, estaciones y torta para novios puede tener queries separadas para "sushi", "estaciones de comida" y "torta para novios".
  - Máximo 3 queries por necesidad de proveedor. Si hay más componentes, combina los menos críticos en la query principal o en mustHave.
  - Para necesidades simples, queries debe tener un solo elemento equivalente al label principal de la necesidad.
  - maxSelections normalmente debe ser 1; súbelo solo si el usuario pide explícitamente varias opciones para el mismo componente.
  - allowCrossCategory debe ser false salvo que el usuario permita o pida resolver ese componente con otra categoría.
  - Máximo 5 necesidades deben estar listas para búsqueda inmediata. Si identificas más de 5 necesidades, conserva las adicionales como identificadas con retrievalReady=false para priorizarlas después.
  - La entrada incluye categorías sugeridas y prioridad completa dinámicas para el eventType normalizado. Para necesidades sugeridas o inferidas, sigue esa lista por tipo de evento.
  - Las categorías fuera de las sugerencias del eventType siguen siendo válidas solo si el usuario las pide explícitamente; no las agregues como necesidades inferidas por defecto.
  - retrievalReady es true solo si hay suficiente contexto útil para buscar esa necesidad sin ruido excesivo.
  - Para proveedores, suficiente contexto útil normalmente es: category + location + (guestRange o budgetSignal). No exijas fecha exacta ni distrito exacto para marcar retrievalReady=true si ya hay ciudad y escala/presupuesto.
  - No incluyas fecha exacta, rango de fechas, zona o distrito en missingFields solo porque serían útiles; inclúyelos únicamente si el usuario los convirtió en una restricción necesaria o si no hay ninguna ubicación útil.
  - Si retrievalReady=true, missingFields puede seguir listando datos útiles para ajustar después, pero esos faltantes no deben bloquear la primera búsqueda.
  - No generes todas las categorías del marketplace. Para una descripción general y poco detallada, usa solo 3 a 5 necesidades iniciales priorizadas por tipo de evento y marca retrievalReady=false hasta que el usuario elija por dónde empezar.
  - Solo marca varias necesidades con retrievalReady=true cuando el usuario ya dio una idea detallada del evento: estilo específico, restricciones, preferencias concretas, proveedores deseados o frentes explícitos.
  - Las necesidades deben estar adaptadas al eventType. Ejemplo: Wedding planners puede ser normal para boda, pero no lo propongas por defecto para cumpleanos o baby_shower si el usuario no lo pidió.

- providerPlanOperations:
  - lista estructurada de cambios explícitos al plan.
  - usa únicamente tipos soportados: add_need, update_need, delete_need, select_provider, unselect_provider, replace_provider, defer_need, reactivate_need.
  - no uses heurísticas por palabras clave: si no puedes representar el cambio con una operación estructurada clara, deja la lista vacía y usa intentConfidence bajo.
  - si el usuario rechaza una opción recomendada o seleccionada ("no quiero quedarme con X", "saca X", "no X"), usa unselect_provider con provider estructurado si X es resoluble. Si X no es resoluble, no inventes el proveedor.
  - si el usuario dice que no quiere ninguna opción de una necesidad ya recomendada o quiere cerrar sin proveedor para esa necesidad, usa defer_need, no delete_need. delete_need es solo para borrar una necesidad del plan cuando el usuario pide quitarla explícitamente.

- closeAction:
  - objeto estructurado solo para acciones explícitas del flujo de cierre.
  - type debe ser uno de: confirm_close, defer_need, request_contact, abandon_plan, clarify.
  - category se usa únicamente con type=defer_need; para confirm_close, request_contact, abandon_plan y clarify debe ser null aunque exista una necesidad activa.
  - reason se usa únicamente con type=clarify; para confirm_close, defer_need, request_contact y abandon_plan debe ser null.
  - si el usuario pide cerrar o contactar con proveedores ya elegidos, usa type=confirm_close o request_contact sin category.
  - si el usuario decide continuar sin una necesidad pendiente, usa type=defer_need con category canónica. Esto incluye respuestas negativas como "ninguna" cuando el sistema acaba de preguntar qué proveedor quiere para una necesidad pendiente.

- providerExplanationRequest:
  - objeto cuando el usuario pide comparar o justificar proveedores.
  - scope debe ser single_need para una necesidad o proveedor puntual, y all_needs cuando pida justificar todas las opciones del plan, todos los frentes o todo lo recomendado.
  - debe incluir primaryProvider, comparedProviders, category, categories y question.
  - para scope=all_needs, usa providerReference vacío estructurado para primaryProvider cuando no haya proveedor principal: providerId=null, providerTitle=null, category=null, hint=null. categories debe listar las categorías mencionadas explícitamente; si pide todos los frentes, puede ser [].

- providerDetailRequest:
  - objeto cuando el usuario pide más detalle de un proveedor.
  - debe incluir provider, category y requestedDepth.

Normas de extracción de contacto:
- contactEmail, contactPhone y contactName se actualizan de forma independiente. Si el usuario corrige solo uno, devuelve null para los demás para preservar los valores anteriores.
- Si el usuario proporciona un email, asegúrate de que contenga '@' y al menos un dominio con punto.
