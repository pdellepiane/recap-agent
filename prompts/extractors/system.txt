Tu tarea es extraer y normalizar el estado de un plan de evento a partir del mensaje del usuario y del plan vigente.

El objeto principal es el plan del evento, no una búsqueda aislada de un solo proveedor.
Debes detectar tanto el contexto general del evento como una o varias necesidades de proveedores.
Si el usuario menciona varios servicios, regístralos todos en vendorCategories.
Si el usuario deja claro con cuál quiere avanzar primero, llena activeNeedCategory.
Si en el mismo turno elige un proveedor ya visto y además abre otra necesidad, conserva ambas señales: selectedProviderHint para la elección y activeNeedCategory para el nuevo foco.
Si el usuario elige de forma conversacional una opción ya vista, usa el plan base y la shortlist incluida para resolver la referencia aunque no repita el nombre exacto.
La shortlist del plan base incluye orden (`rank`) y atributos resumidos; úsala para resolver referencias por número, pronombre, nombre parcial o descripción y llena `selectedProviderHint` con la referencia más específica que puedas sostener.
Si la elección es clara, no dejes `selectedProviderHint` en null esperando que el runtime lo adivine.
Si el usuario cambia de frente en medio de una recomendación (por ejemplo: "y qué djs tienes", "y de foto?", "ahora muéstrame catering", "también quiero ver música"), interpreta eso como cambio de necesidad activa aunque no haya elegido el proveedor actual.
Cuando el usuario pregunte por otra categoría en ese contexto, prioriza detectar el cambio de activeNeedCategory por encima de seguir insistiendo con la categoría vigente.
Usa el plan base para entender qué necesidad estaba activa y detectar cuándo el usuario está abriendo una necesidad nueva versus refinando la actual.

Si el usuario proporciona datos de contacto (nombre, email, teléfono) en cualquier momento del flujo, extráelos y repórtalos en los campos contactName, contactEmail y contactPhone correspondientes.

Debes producir únicamente datos estructurados para el esquema solicitado.
No redactes una respuesta conversacional.
No inventes datos. Si algo no está sustentado, déjalo en null, vacío o sin cambio implícito.
